# 생성 품질 테스트 케이스 (개발자용)

> **대상 독자:** 개발자(나). 배포 전 생성 품질을 반복 검증하기 위한 체크리스트다.
> 사용자에게 보여주는 랜딩 문구가 아니다.
>
> **목적:** 앱 유형별 입력을 고정해두고, 코드 수정 후 "생성 결과가 의도대로 나오는지"를
> 매번 같은 기준으로 빠르게 확인한다.

---

## 0. 사전 지식: 생성 파이프라인 4단계

입력 → 결과까지 거치는 단계와 각 단계를 담당하는 함수:

| 단계 | 함수 | 위치 | 출력 |
|------|------|------|------|
| ① 질문 placeholder | `generateQuestions` / `detectPlaceholders` | `src/lib/mock-ai.ts` | Q&A 입력 힌트 |
| ② 기획서 | `generatePlan` (`generateAppName`, `extractFeatures`) | `src/lib/mock-ai.ts` | `appName`, `purpose`, `coreFeatures`, `techStack` |
| ③ 화면 목록 | `generateScreens` | `src/lib/mock-ai.ts` | `Screen[]` |
| ④ 미리보기/코드 | `generateApp` (`detectAppType`, 섹션 빌더) → `schemaToTsx` → ZIP | `src/lib/mock-ai.ts`, `src/lib/schema-to-tsx.ts`, `src/components/preview/CodeViewModal.tsx` | `UISchema` → TSX → ZIP |

### 알아둘 핵심 동작 (테스트 판정의 근거)

1. **유형 분류(`detectAppType`)는 "먼저 매칭되는 키워드"가 이긴다.**
   순서: `marketplace → finance → todo → booking → fitness → learning → schedule → food → community → default`.
   예: "동네 중고거래"는 `중고`(marketplace)가 `동네`(community)보다 먼저라 **marketplace**로 분류됨.

2. **앱 이름(`generateAppName`)은 유형과 별개의 규칙(`APP_NAME_RULES`)으로 결정된다.**
   유형은 맞아도 이름 규칙이 없으면 `나만의 앱`으로 폴백된다. (todo·booking 일부 입력에서 발생 → 아래 케이스에서 플래그)

3. **화면 목록(`generateScreens`)은 유형이 아니라 `coreFeatures`로 결정된다.**
   `extractFeatures`가 뽑은 기능 키워드에 따라 화면이 추가된다. 항상 `홈` + `설정`은 포함.

4. **미리보기 섹션만 유형별 빌더를 쓴다.** 즉 화면 목록과 미리보기 섹션은 서로 다른 로직이라 따로 봐야 한다.

## 1. 공통 확인 항목 (모든 케이스에 적용)

각 테스트 케이스마다 아래 6개를 확인한다.

- [ ] **C1. 앱 이름이 자연스러운가** — `나만의 앱` 폴백이 아니라 의도에 맞는 이름인가
- [ ] **C2. 기획서가 입력 의도와 맞는가** — `purpose`, `targetUser`, `coreFeatures`가 설명·답변을 반영하는가
- [ ] **C3. 화면 목록이 앱 유형과 맞는가** — `coreFeatures`로 추가된 화면이 적절한가 (홈/설정 외)
- [ ] **C4. 미리보기 섹션 구성이 적절한가** — 분류된 유형의 섹션 빌더가 의도와 맞는가
- [ ] **C5. ZIP 다운로드 후 실행 가능한가** — `npm install && npm run dev` → `localhost:3000` 정상 렌더
- [ ] **C6. ZIP 내부 동작이 안 깨지는가** — Form 저장/Chat 전송/List 펼침/CardGrid 모달 + 새로고침 후 유지

> C5·C6은 케이스마다 매번 돌리기 무거우면 **유형당 1개 대표 케이스**만 전체 실행하고, 나머지는 C1~C4(생성 결과만)로 빠르게 훑는다.

## 2. 유형별 테스트 입력

각 케이스의 **입력 설명**과 **Q&A 답변**을 그대로 복사해 넣고 결과를 비교한다.
`기대 유형` / `기대 이름` / `기대 테마색`은 현재 코드 기준 예상값이다 (틀리면 그게 곧 회귀 신호).

### TC-01 · marketplace (중고거래/마켓)

- **입력 설명:** `동네 주민들과 중고 물건을 사진 찍어 올리고 채팅으로 거래하는 앱`
- **Q&A 답변:**
  - q1: `동네에 사는 20~40대 주민`
  - q2: `사진을 올려서 중고 물건 팔기`
  - q3: `내 주변에 올라온 중고 물건 목록`
  - q4: `물건 사진 올리기, 판매자와 채팅하기`
  - q5: `깔끔하고 신뢰감 있는`
- **기대 유형:** `marketplace` (`중고`)
- **기대 이름:** `동네마켓` (`중고`+`동네`)
- **기대 테마색:** `#10B981` / minimal
- **기대 미리보기 섹션:** Hero → List(최근 상품) → Form(상품 등록) → **Chat(판매자와 채팅)**
- **포인트 체크:**
  - [ ] Chat 섹션이 생성됨 → ZIP에서 **메시지 전송 + 새로고침 유지** 반드시 확인 (Chat은 marketplace에만 있음)
  - [ ] 화면 목록에 `채팅`, `검색`, `결제` 등 기능 화면이 들어왔는지

## 4. ZIP 실행/동작 검증 절차 (C5·C6 상세)

대표 케이스 ZIP을 받은 뒤:

```bash
# 1. 압축 해제 후 폴더 진입
npm install
npm run dev
# 2. http://localhost:3000 접속
```

### C5. 실행 가능 체크
- [ ] `npm install` 에러 없음 (next 15 / react 19 의존성)
- [ ] `npm run dev` 부팅 성공, 콘솔 에러 없음
- [ ] 페이지 렌더 — 헤더(앱 이름) + 모든 섹션 표시
- [ ] (선택) `npm run build` 통과 — 타입 에러 없는지

### C6. 내부 컴포넌트 동작 체크
ZIP 내부 `src/app/components/GeneratedApp.tsx` 기준:

- [ ] **Form** — 입력 후 제출 → "저장되었습니다 ✓" + 하단 "저장된 항목" 목록에 추가 → **새로고침 후에도 유지** (`localStorage` 키 `mwab_{slug}_form_{idx}`)
- [ ] **Chat** (marketplace 전용) — 메시지 입력/Enter 전송 → 우측 말풍선 추가, 자동 스크롤 → **새로고침 후 유지** (`mwab_{slug}_chat_{idx}`)
- [ ] **List** — 항목 클릭 시 아래 상세 패널 펼침, 다시 클릭/× 로 닫힘
- [ ] **CardGrid** — 카드 클릭 시 모달, × 또는 배경 클릭으로 닫힘
- [ ] 여러 Form/List가 있을 때 각각 **독립적인 state/storage 키**를 갖는가 (`idx`로 구분)

> **localStorage 주의:** 같은 브라우저에서 여러 ZIP 앱을 `localhost:3000`으로 띄우면 slug가 같을 경우 키가 충돌할 수 있다. 케이스 전환 시 DevTools → Application → Local Storage 비우고 테스트.

## 5. 빠른 회귀 실행 순서 (권장)

1. **생성만 빠르게 (C1~C4):** TC-01 ~ TC-10 입력을 순서대로 넣고 유형/이름/테마/섹션 표와 대조

## 6. 알려진 갭 / 추적 항목

코드 수정 시 함께 고려할 후보 (지금은 문서로만 추적, 코드 미수정):

- **todo·일부 입력의 이름 폴백** — `APP_NAME_RULES`에 `할일/습관/체크리스트/루틴/투두` 규칙 없음 → `나만의 앱`. (TC-03)
- **유형 분류 우선순위 의존** — 복합 의도 입력에서 먼저 오는 키워드가 이김. 의도와 다르면 `APP_TYPE_KEYWORDS` 순서/키워드 조정 필요. (3절)
- **화면 목록 vs 미리보기 섹션 불일치 가능성** — 둘은 다른 로직(`coreFeatures` vs `appType`). 유형은 맞는데 화면 목록이 빈약할 수 있음 → C3와 C4를 분리해 본다.
